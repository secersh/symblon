## undercover — webhook ingestion pipeline (2026-03-09)

### Folder structure
- Follow Go standard project layout: `cmd/`, `internal/`, `pkg/`
- `pkg/` for packages shared across binaries (messaging, event model)
- `internal/` for packages private to this module (handlers, normalizers, writer)

### Webhook event model
- Single provider-agnostic `WebhookEvent` struct in `pkg/event/`
- Adding a new provider requires only a new normalizer + a `Provider` constant; the rest of the pipeline is unchanged
- Original raw payload preserved in the `Payload` field for audit and re-processing

### Messaging
- RabbitMQ topic exchange `symblon` — routing key convention: `activity.<provider>`
- Consumers bind with AMQP wildcard patterns (e.g. `activity.#`) to receive events from all providers
- Both ingestor and paquetier declare the exchange idempotently on startup

### Storage
- DuckDB in-process instance buffers events in memory; flushed to Parquet every 30 s and on shutdown
- Hive-style partition layout: `provider=X/year=Y/month=M/day=D/event_type=E/`
- Enables partition pruning by any Parquet-aware engine (DuckDB, Spark, Athena)

### Local development
- Multi-stage Dockerfile with CGO support (gcc/musl-dev) for DuckDB; separate `ingestor` and `paquetier` build targets
- `docker-compose.yml` orchestrates RabbitMQ, ingestor, and paquetier; all services wait on RabbitMQ healthcheck
- Opt-in Cloudflare Tunnel (Docker Compose profile `tunnel`) for receiving real webhooks locally without port-forwarding
